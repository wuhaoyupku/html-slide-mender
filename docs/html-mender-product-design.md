# HTML Mender 产品设计稿

## 1. 产品定位

HTML Mender 是一个浏览器插件，用来把 AI 生成的 HTML PPT、静态演示页面、网页化报告变成可直接修稿的页面。

它不把页面导入一个新的编辑器，也不缩小原页面。用户仍然在浏览器里以原始比例查看页面，插件只在页面上方叠加一层轻量编辑能力。

一句话定位：

> 打开 HTML PPT，点击插件，当前页面里的文字和图片立刻可编辑，改完下载新的 HTML。

## 2. 核心原则

### 2.1 不接管导航

用户翻到哪一页，就编辑哪一页。

插件不做左侧缩略图，不重写原页面的翻页逻辑，不要求用户进入一个独立的“设计工作台”。原页面本来怎么翻页，就继续怎么翻页。

插件只需要做到：

- 识别当前可见区域里的可编辑文字和图片。
- 当用户翻页后，自动重新识别当前可见内容。
- 用户修改的内容能进入最终下载的 HTML。

### 2.2 不改变原页面比例

编辑时页面仍然保持 `100vw × 100vh` 或原网站当前布局。

不把 PPT 页面缩小到画布里，不加固定侧栏挤压内容，不改变 body、deck、slide 的布局尺寸。

### 2.3 只做内容修稿，不做复杂排版

MVP 只编辑：

- 文字内容
- 文字样式
- 图片替换
- 图片在原容器内的缩放、移动、适配

不做：

- 新增页面
- 删除页面
- 重排布局
- 跨页管理
- 复杂动画编辑
- 组件层级编辑
- flex/grid 布局重构

### 2.4 插件是浮层，不是页面的一部分

编辑工具栏、选框、控制点、快捷菜单都属于插件浮层。

保存时必须自动移除这些插件 UI，只保留用户真正修改过的 HTML 内容和样式。

## 3. 目标用户

### 3.1 普通内容用户

他们会用 AI 生成 HTML PPT，但不会改 HTML/CSS。

典型需求：

- 改错别字。
- 换标题。
- 改字号和颜色。
- 替换配图。
- 改完直接发给别人。

### 3.2 AI PPT 工具使用者

他们经常用模型生成网页式 PPT，但需要快速微调。

典型需求：

- 不回到提示词里反复生成。
- 不找工程师帮忙改代码。
- 快速完成“最后 10%”修稿。

### 3.3 内容交付者

例如咨询、培训、销售、创业者、内容创作者。

典型需求：

- 客户说“这里换个词”，能现场改。
- 演讲前发现图不对，能快速替换。
- 不破坏原版式。

## 4. 核心场景

### 场景 A：修改 AI HTML PPT 文案

1. 用户打开本地或线上 HTML PPT。
2. 点击插件。
3. 页面进入编辑模式。
4. 当前页文字区域出现浅蓝色可编辑框。
5. 用户点击标题，直接修改文字。
6. 使用顶部工具栏改字体、字号、颜色。
7. 点击保存，下载新 HTML。

### 场景 B：替换当前页图片

1. 用户打开某一页。
2. 点击插件进入编辑模式。
3. 图片出现浅绿色可编辑框。
4. 用户点击图片。
5. 浮动图片工具条出现。
6. 用户点击替换，上传本地图片。
7. 新图片进入原容器。
8. 用户缩放、拖动，让画面适配。
9. 点击保存。

### 场景 C：编辑普通网页快照

1. 用户打开一个普通网页。
2. 点击插件。
3. 插件识别当前可见区域里的文本和图片。
4. 用户做本地修改。
5. 插件导出修改后的 HTML 快照。

注意：普通网站不承诺回写原网站，只承诺导出本地快照。

## 5. 产品结构

MVP 由四个可见 UI 层组成：

1. 插件入口
2. 顶部浮动主工具栏
3. 元素选框与就地快捷工具条
4. 保存面板

### 5.1 插件入口

浏览器右上角插件图标。

点击后弹出小面板：

- 启动编辑
- 最近保存状态
- 当前页面是否可编辑
- 权限提示

第一版面板可以非常轻：

```text
HTML Mender

当前页面可编辑
[开始编辑]

仅在你点击后修改当前页面。
```

进入编辑模式后，再次点击插件图标可显示：

```text
HTML Mender

正在编辑当前页面
[显示 / 隐藏编辑框]
[下载 HTML]
[退出编辑]
```

### 5.2 顶部浮动主工具栏

位置：

- 固定在浏览器页面顶部居中。
- 不占页面布局空间。
- 默认半透明。
- 鼠标移入变实。
- 可折叠成一个小胶囊。

内容：

```text
HTML Mender | 文本 | 图片 | 撤销 | 重做 | 显示编辑框 | 下载 HTML | 退出
```

当选中文字时，工具栏切换为文字工具：

```text
字体  字号  B  I  U  颜色  高亮  左对齐  居中  右对齐  行高
```

当选中图片时，工具栏切换为图片工具：

```text
替换图片  填满  适应  放大  缩小  重置  删除修改
```

设计原则：

- 工具栏始终浮在页面之上。
- 不改变原页面布局。
- 不做 Office 全量功能，只放最常用修稿能力。

### 5.3 元素选框

进入编辑模式后，插件给当前可见内容添加编辑提示框。

文字框：

- 默认浅蓝虚线。
- hover 时蓝色实线。
- 选中时蓝色实线加控制点。

图片框：

- 默认浅绿色虚线。
- hover 时绿色实线。
- 选中时绿色实线加控制点。

选框不写入原 HTML 文件，只作为插件 UI。

### 5.4 就地快捷工具条

选中元素后，在元素上方或下方出现小型快捷条。

文字快捷条：

```text
编辑  字号 - / +  颜色  加粗
```

图片快捷条：

```text
替换  缩放  拖动  重置
```

如果顶部空间不足，快捷条自动出现在元素下方。

如果元素靠近屏幕边缘，快捷条自动向内避让。

### 5.5 保存面板

点击下载 HTML 后弹出保存面板：

```text
保存修改

已修改：
文字 8 处
图片 2 处
样式 5 处

[下载单文件 HTML]
[复制当前 HTML]
[取消]
```

第一版主路径是“下载单文件 HTML”。

## 6. 编辑模式状态

### 6.1 未启动

页面保持原样。

插件不注入编辑框，不监听键盘，不修改 DOM。

### 6.2 扫描中

用户点击“开始编辑”后，顶部显示：

```text
正在识别当前页面内容...
```

扫描当前可见区域中的文字和图片。

### 6.3 可编辑

可编辑元素显示边框。

用户可以选中文字和图片。

原页面导航保持可用，但插件会尽量避免工具栏遮挡原页面导航按钮。

### 6.4 正在编辑文字

当前文字块进入 `contenteditable` 状态。

插件暂停页面自身的快捷键响应，避免用户输入空格、方向键时触发翻页。

退出文本编辑后，恢复页面快捷键。

### 6.5 正在编辑图片

当前图片进入图片编辑状态。

用户可以：

- 替换
- 拖动画面位置
- 缩放
- 切换适应 / 填满
- 重置

图片容器不变。

### 6.6 保存中

插件生成干净 HTML：

- 移除插件工具栏。
- 移除选框。
- 移除临时属性。
- 保留用户修改。
- 内嵌上传图片。

### 6.7 已保存

下载完成后显示轻提示：

```text
已下载修改后的 HTML
```

用户仍可继续编辑当前页面。

## 7. 文字编辑规格

### 7.1 可识别文字元素

优先识别：

- `h1` 到 `h6`
- `p`
- `span`
- `li`
- `blockquote`
- `button`
- `a`
- 含有直接文本内容的 `div`
- 带 `data-editable` 的任意元素

排除：

- `script`
- `style`
- `noscript`
- `svg`
- `canvas`
- 隐藏元素
- 插件自身 UI
- 宽高过小的文本碎片
- 纯装饰字符

### 7.2 文字选中逻辑

单击文字框：

- 选中整个文字块。
- 显示文字工具栏。

双击文字框：

- 进入直接编辑。
- 光标定位到点击位置。

直接输入：

- 如果文字块已选中，用户输入字符时自动进入编辑。

Esc：

- 退出当前文字编辑。

Cmd / Ctrl + Z：

- 撤销最近一次插件编辑。

### 7.3 文字样式

MVP 支持：

- 字体
- 字号
- 字重
- 斜体
- 下划线
- 文字颜色
- 高亮色
- 对齐
- 行高

默认操作规则：

- 如果用户没有选择局部文字，样式作用于整个文字块。
- 如果用户选择了一段文字，样式作用于选区。
- 如果局部样式实现复杂，第一版可以先统一作用于整个文字块。

### 7.4 字体选择

字体下拉包含：

- 当前字体
- 系统默认
- Arial
- Helvetica
- Inter
- Noto Sans SC
- Noto Serif SC
- 思源黑体
- 思源宋体
- 苹方
- 微软雅黑

如果某字体当前设备没有安装，浏览器会按 CSS fallback 处理。

### 7.5 字号

字号控件包括：

- 数值输入
- 加号
- 减号

单位默认使用当前元素已有单位。

如果无法判断，使用 `px`。

### 7.6 文本溢出保护

用户改文字后，插件不主动重排页面，但要提示溢出。

当文字内容超出原元素边界时：

- 文字框变橙色。
- 快捷条显示“内容可能溢出”。
- 提供“一键缩小字号”。

第一版不自动改布局。

## 8. 图片编辑规格

### 8.1 可识别图片元素

优先识别：

- `img`
- `picture img`
- 带 CSS `background-image` 的元素
- 带 `data-image-slot` 的元素

暂不处理：

- canvas 绘制图片
- 跨域 iframe 内图片
- 视频帧
- SVG 内复杂图片结构

### 8.2 图片选中逻辑

单击图片框：

- 选中图片。
- 显示图片工具栏。

双击图片：

- 进入图片调整模式。

拖拽上传：

- 用户把图片拖到当前图片框上，即替换图片。

### 8.3 图片替换

点击“替换图片”后：

1. 打开文件选择器。
2. 用户选择本地图片。
3. 插件读取图片。
4. 替换当前图片内容。
5. 保持原容器尺寸不变。

支持格式：

- PNG
- JPG / JPEG
- WebP
- GIF 第一版可作为静态资源保留，不做动画编辑。

### 8.4 图片适配方式

提供三种模式：

- 填满：`object-fit: cover`
- 适应：`object-fit: contain`
- 拉伸：`object-fit: fill`

默认替换后使用“填满”，因为大多数 PPT 图片槽需要视觉铺满。

### 8.5 图片缩放和拖动

对于 `img`：

- 外层容器不动。
- 图片本身使用 `transform: translate(...) scale(...)`。
- 容器使用 `overflow: hidden`。

对于 `background-image`：

- 使用 `background-size` 控制缩放。
- 使用 `background-position` 控制位置。

### 8.6 图片重置

重置只撤销当前图片的插件编辑：

- 恢复原图片。
- 恢复原 fit 模式。
- 恢复原位置和缩放。

## 9. 保存与导出

### 9.1 MVP 保存目标

第一版只承诺：

> 下载一个修改后的新 HTML 文件。

不承诺：

- 覆盖原本地文件。
- 回写线上网站。
- 更新服务器内容。
- 同步到 AI 生成工具后台。

### 9.2 导出内容

导出的 HTML 包含：

- 当前页面完整 HTML。
- 用户修改后的文本。
- 用户修改后的样式。
- 用户替换后的图片。
- 原页面 CSS 和 JS。

导出的 HTML 不包含：

- 插件工具栏。
- 编辑框。
- 控制点。
- 插件临时 class。
- 插件事件监听脚本。

### 9.3 图片内嵌

用户上传的新图片默认转成 base64 data URL。

优点：

- 下载后是单文件。
- 发给别人也能打开。
- 不依赖本地图片路径。

缺点：

- 文件会变大。

后续版本可提供“图片单独打包 ZIP”。

### 9.4 保存前预检查

点击下载时，保存面板提示：

- 修改数量
- 是否存在文字溢出
- 是否存在未加载图片
- 文件预计大小

如果有溢出：

```text
有 2 个文字块可能溢出，仍然下载？
[继续下载] [返回修改]
```

## 10. 当前页识别机制

虽然不做导航，但插件需要知道“当前看得见什么”。

识别方式：

- 通过 `IntersectionObserver` 找当前视口内可见元素。
- 只给可见区域内的元素显示编辑框。
- 用户翻页、滚动、resize 后，自动重新扫描。
- 如果页面是 transform 横向 deck，仍以视口交集判断可见元素。

插件不需要理解所有 PPT 框架。

只要当前元素真实出现在视口里，就可以编辑。

## 11. 交互细节

### 11.1 显示 / 隐藏编辑框

顶部工具栏提供“显示编辑框”开关。

关闭后：

- 隐藏所有浅色边框。
- hover 时仍显示可编辑提示。
- 当前选中元素仍显示选框。

适合用户检查最终画面。

### 11.2 暂停页面快捷键

当用户正在输入文字时：

- 插件拦截空格、方向键、Enter 等输入。
- 防止原 HTML PPT 翻页。

当用户不在文字编辑状态：

- 原页面快捷键继续生效。

### 11.3 链接编辑

如果点击的是链接文本：

- 第一次点击选中链接文字。
- 不跳转。
- 工具条显示“打开链接”小按钮。

这样避免用户编辑时误跳转。

### 11.4 工具栏避让

顶部工具栏可能遮住页面标题。

解决方式：

- 默认半透明。
- 可以拖动到顶部、底部、左上、右上四个停靠点。
- 可以折叠成小胶囊。

MVP 至少支持折叠。

### 11.5 原页面动效

插件不编辑动画。

如果页面有入场动画：

- 插件等动画完成后再扫描。
- 用户也可以点击“重新识别当前页”。

## 12. 权限与安全感设计

插件权限文案要克制：

```text
仅在你点击插件后，临时编辑当前标签页。
不会上传你的页面内容。
不会自动保存到网站服务器。
```

MVP 尽量只需要：

- 当前标签页权限
- 注入脚本权限
- 下载权限
- 本地临时存储权限

所有编辑默认发生在本地浏览器。

## 13. 技术产品约束

### 13.1 适合的页面

体验最好：

- AI 生成 HTML PPT
- 静态 HTML slides
- 单文件演示页
- 本地 HTML 文件
- 静态落地页

体验一般：

- 普通响应式网站
- React / Vue 应用
- 有复杂状态刷新的页面

不适合：

- canvas PPT
- PDF
- 跨域 iframe
- 高强度交互 Web App
- 需要登录态服务端保存的网站后台

### 13.2 兼容性策略

优先支持：

- Chrome
- Edge
- Brave
- Arc

这些都是 Chromium 插件生态，第一版落地成本最低。

后续适配：

- Firefox

Firefox 可能需要单独处理部分 Manifest V3 和文件 API 差异。

## 14. MVP 功能清单

### 必须有

- 插件启动编辑模式
- 当前可见文本识别
- 当前可见图片识别
- 文字选框
- 图片选框
- 文字直接编辑
- 字体、字号、颜色、加粗
- 图片替换
- 图片填满 / 适应
- 下载修改后的 HTML
- 退出编辑模式

### 应该有

- 撤销 / 重做
- 显示 / 隐藏编辑框
- 图片缩放
- 图片拖动
- 文字溢出提示
- 保存前修改统计

### 可以后置

- 局部文字选区样式
- 图片裁剪面板
- 拖动文字框
- 调整文字框尺寸
- 页面缩略图
- AI 改写
- 多文件 ZIP 导出
- 覆盖保存本地 HTML

### 明确不做

- 不做页面导航系统
- 不做左侧缩略图栏
- 不做完整图层面板
- 不做新增/删除 slide
- 不做复杂布局编辑
- 不做线上网站回写

## 16. 成功标准

MVP 成功不是“能编辑所有网页”，而是：

- 用户可以在 10 秒内进入编辑模式。
- 用户能一眼看出哪些文字和图片可编辑。
- 用户可以在不看代码的情况下改完一页。
- 修改不会破坏原页面布局。
- 导出的 HTML 能独立打开。
- 编辑框和工具栏不会出现在导出文件里。

## 17. 推荐迭代路线

### V0.1 内容修稿版

- 当前可见图片替换
- 顶部浮动工具栏

### V0.2 稳定可用版

- 一键统一字体
- 一键统一品牌色
- 图片替换建议

## 18. 最小可开发版本定义

第一版可以只做一个按钮：

> 点击插件后，当前页面所有可见文字和图片出现可编辑框。

再做三个动作：

1. 文字能改。
2. 图片能换。

只要这三个动作稳定，这个产品就已经有价值。

## 19. 第一版开发任务拆分

### 19.1 插件壳

交付内容：

- 浏览器工具栏图标。
- Popup 面板。
- “开始编辑”按钮。

### 19.2 编辑浮层

交付内容：

- 可折叠。
- 不影响原页面布局。

- 导出的 HTML 中不包含工具栏 DOM。

- 识别当前视口可见文本。
- 识别当前视口可见图片。
- 给文字和图片绘制选框。
- 翻页或滚动后自动重新识别。

验收标准：

- 用户当前看到的主要标题、正文、图片都有选框。
- 隐藏页面或屏幕外页面不显示选框。
- 插件自身 UI 不会被识别为可编辑内容。

### 19.4 文字编辑

交付内容：

- 点击选中文字块。
- 双击进入编辑。
- 修改文字内容。
- 修改字体、字号、颜色、加粗。
- 退出编辑后保留修改。

验收标准：

- 修改标题和正文不触发原页面翻页。
- 改完后原页面视觉位置基本不变。
- 下载 HTML 后文字修改仍然存在。

### 19.5 图片编辑

交付内容：

- 点击选中图片。
- 替换本地图片。
- 填满 / 适应模式切换。
- 基础缩放。
- 重置当前图片修改。

验收标准：

- 新图片在原图片区域内显示。
- 原容器尺寸不变。
- 下载 HTML 后替换图片仍可显示。

### 19.6 导出保存

交付内容：

- 统计修改数量。
- 移除插件 UI。
- 内嵌新图片。
- 生成并下载新 HTML。

验收标准：

- 下载文件可以直接双击打开。
- 打开后没有编辑框和工具栏。
- 用户修改过的文字、样式、图片都存在。
- 原页面主要 CSS 和 JS 仍然工作。

## 20. 设计决策记录

### 20.1 为什么不做导航

因为 HTML PPT 本身通常已经有导航逻辑。

插件如果重做导航，会遇到三个问题：

- 侵入原页面，容易破坏现有翻页。
- 占据屏幕空间，影响用户按原比例修稿。
- 产品心智会变成完整 PPT 编辑器，第一版复杂度失控。

所以第一版只做“当前可见内容编辑”，不做“页面管理”。

### 20.2 为什么不做侧栏

侧栏会压缩原页面，导致用户看到的不是最终效果。

HTML PPT 的价值在于全屏演示视觉，编辑也应该保持全屏视觉。

所以工具采用浮动模式，而不是固定侧栏模式。

### 20.3 为什么先做下载 HTML

覆盖保存和网站回写都涉及权限、文件系统、服务器 API 或跨浏览器兼容。

下载新 HTML 是最稳定、最容易解释、最符合用户安全感的路径。

### 20.4 为什么只改图片内部，不改容器

容器通常决定页面布局。

只要容器不动，页面不容易崩。

所以第一版图片编辑只处理：

- 图片资源
- 图片适配方式
- 图片在容器内的位置和缩放

不处理容器位置、尺寸和层级。
